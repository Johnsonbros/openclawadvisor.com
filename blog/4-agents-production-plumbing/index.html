I own Johnson Bros Plumbing in Boston. We’re not a tech company. We fix water heaters, unclog drains, and install toilets. But for the past 18 months, we’ve been running a 4-agent AI system built on OpenClaw that handles scheduling, customer communications, research, and daily operations.

This isn’t a demo. This is production. Real customers. Real money. Real stakes.

And because a security breach could kill my business, we had to get security right. This is the full story: why we built it, how it works, what we learned, and how we hardened it.

Why We Built This#

In 2024, I hit a wall. My plumbing business was growing, but I couldn’t keep up. I was drowning in:

  • Customer calls and texts at all hours
  • Scheduling conflicts and double-bookings
  • Research questions (“What’s the best tankless water heater for a 3-bath house?”)
  • Administrative work‚Äîinvoices, follow-ups, ordering parts

The obvious answer: hire more people. But that’s expensive, hard to manage, and I’d still be the bottleneck for decisions.

Then OpenClaw hit my Twitter feed. An AI assistant that could do things‚Äînot just answer questions. It could send messages, search the web, run scripts, integrate with tools. I thought: What if I could automate the coordination layer of my business?

So I built it. And it worked. Too well, actually‚Äîwhich is when I realized how insecure it was.

The Architecture#

Our system runs on a dedicated Ubuntu server (Hetzner, $40/month) with these components:

Infrastructure#

  • OpenClaw Gateway (main process)
  • Docker containers for each agent (isolation)
  • PostgreSQL (conversation history and knowledge base)
  • Redis (job queue and caching)
  • Nginx (reverse proxy with SSL)
  • HashiCorp Vault (secret management)

Communication Channels#

  • Telegram (my primary interface)
  • SMS (customer communications via Twilio)
  • Voice calls (for urgent customer requests)
  • Email (vendor communications)

Security Layers#

  1. Container isolation ‚Äî each agent runs in its own Docker container
  2. Filesystem restrictions ‚Äî read-only mounts except for specific directories
  3. Network segmentation ‚Äî agents can’t access each other directly
  4. Tool permissions ‚Äî strict whitelisting of allowed commands
  5. Approval workflows ‚Äî high-risk actions require my confirmation
  6. Secret rotation ‚Äî API keys and credentials rotate every 90 days
  7. Monitoring and alerts ‚Äî unusual activity triggers notifications

Security Lessons Learned#

Lesson 1: Default OpenClaw Is Not Production-Ready#

When I first deployed, I ran OpenClaw with default settings. Within a week, I realized:

  • WebSocket ports were exposed to the internet
  • Agents had unlimited filesystem access
  • API keys were in plain text config files
  • No audit logging of agent actions

This was terrifying. One prompt injection attack away from disaster.

Lesson 2: Container Isolation Is Non-Negotiable#

Running all 4 agents in one process meant a compromised agent could access everything. Docker containers with strict mounts solved this.

Each agent only sees:

  • Its own conversation history
  • Approved tool directories
  • Shared read-only knowledge base

Lesson 3: Tool Permissions Must Be Explicit#

OpenClaw’s exec tool can run any shell command by default. We locked it down:

  • Whitelist of allowed commands (grep, curl, specific scripts)
  • No shell access (direct execution only)
  • Input sanitization (block injection attempts)
  • Logging of all executions

Lesson 4: Human-in-the-Loop for High Stakes#

We implemented approval workflows for:

  • Scheduling new customers (in case of spam/fraud)
  • Spending over $500 (ordering expensive parts)
  • System changes (modifying automations)
  • Unusual requests (anything that doesn’t fit patterns)

This keeps me in control while automating 90% of routine work.
