1. Session Management Is Your Secret Weapon#

What beginners do: Run everything in one long conversation session

What power users know: Strategic session management is the key to agent reliability

The Problem with Long Sessions#

Context windows have limits. Even with 200K token models, long sessions suffer from:

  • Context dilution ‚Äî important instructions get buried
  • Drift ‚Äî agent behavior shifts over time
  • Token cost ‚Äî you’re paying for thousands of irrelevant tokens
  • Error accumulation ‚Äî small mistakes compound

The Power User Approach: Session Scoping#

Create task-specific sessions with clear boundaries:

# Bad: Everything in one session

openclaw chat --agent main

# Good: Scoped sessions by task
openclaw chat --agent main --session "customer-support-$(date +%Y%m%d)"
openclaw chat --agent research --session "project-alpha-research"
openclaw chat --agent automation --session "weekly-report-generation"

Benefits:

  • Fresh context for each task
  • Easier debugging (isolated logs)
  • Predictable behavior
  • Lower token costs

Session Patterns We Use#

1. Daily Sessions

  • Customer support gets a new session each day
  • Previous day’s insights summarized and injected into new session

2. Task-Based Sessions

  • Each research project gets its own session
  • Automation workflows run in dedicated sessions

3. Ephemeral Sessions

  • One-off questions use temporary sessions
  • Automatically cleaned up after 24 hours

2. Tool Permissions Are More Important Than Prompts#

What beginners focus on: Writing the perfect system prompt

What power users know: Tool permissions determine what’s actually possible

The Hierarchy of Control#

  1. Infrastructure (Docker, network, filesystem) ‚Äî what the environment allows
  2. Tool permissions (which tools, what scope) ‚Äî what actions are possible
  3. System prompts (instructions, guidelines) ‚Äî how the agent should behave
  4. User prompts (requests, questions) ‚Äî what the user wants

Your prompts are suggestion. Your tool permissions are law.

Practical Example#

Instead of prompting “Never delete files,” configure the write tool to be read-only:

[tools.write]
enabled = true
allowed_paths = ["/data/output", "/tmp"]
read_only = true  # Can only create, not delete or modify

Instead of prompting “Only search approved websites,” whitelist domains:

[tools.browser]
enabled = true
allowed_domains = ["wikipedia.org", "company-docs.com"]
block_mode = "deny-all-except-allowed"

Result: Agent can’t violate constraints even if prompted to.

3. Structured Outputs > Natural Language Responses#

What beginners expect: Conversational responses from agents

What power users demand: Structured, parseable output

The Problem with Natural Language#

When agents respond conversationally:

  • Hard to parse programmatically
  • Inconsistent formats
  • Error-prone extraction
  • Wastes tokens on filler

Power User Pattern: Force JSON Output#

For agent-to-agent communication and automation workflows, use structured outputs:

# Bad: Natural language parsing
response = agent.ask("What's the customer's phone number?")
# Response: "The customer's phone number is 555-1234."
phone = extract_phone_somehow(response)  # fragile!

# Good: Structured output
response = agent.ask(
    "Extract customer contact info",
    output_schema={
        "phone": "string",
        "email": "string",
        "name": "string"
    }
)
# Response: {"phone": "555-1234", "email": "...", "name": "..."}
phone = response["phone"]  # reliable!

Our Standard Schemas#

We have JSON schemas for:

  • Customer information extraction
  • Appointment scheduling
  • Research report formatting
  • Decision logging

This makes automation reliable and testable.

4. Memory Management Beats Prompt Engineering#

What beginners do: Craft elaborate system prompts with all context

What power users do: Build external memory systems

The Limit of In-Context Learning#

Even with huge context windows, in-prompt knowledge has problems:

  • Expensive (tokens cost money)
  • Static (doesn’t update easily)
  • Unstructured (hard to query precisely)
  • Overwhelming (agent attention dilutes)

Power User Approach: External Knowledge Bases#

Build structured memory systems:

Vector databases (for semantic search):

# Store past customer interactions
vectordb.store(
    content="Customer John reported water heater issues in Dec 2025",
    metadata={"customer": "John", "date": "2025-12-15", "type": "service_call"}
)

# Later, agent queries:
results = vectordb.search("John's previous issues")

Relational databases (for structured data):

-- Track all appointments
CREATE TABLE appointments (
    id SERIAL PRIMARY KEY,
    customer_name TEXT,
    service_type TEXT,
    scheduled_date TIMESTAMP,
    agent_notes TEXT
);

File-based knowledge (for documentation):

The Retrieval-Augmented Generation (RAG) Pattern#

  1. User asks question
  2. Agent searches knowledge base
  3. Retrieves relevant context
  4. Generates answer using retrieved context

Benefits:

  • Always up-to-date (add new knowledge anytime)
  • Scalable (unlimited knowledge, constant token cost)
  • Auditable (know which sources informed each answer)

5. Approval Workflows Are Non-Negotiable for Production#

What beginners assume: Agents should be fully autonomous

What power users know: Human-in-the-loop prevents disasters

The Spectrum of Autonomy#

Not every decision needs the same level of oversight:

LowNone (auto-execute)Answering FAQs, logging data, scheduling standard appointments

We use a simple approval queue system:

    def request_approval(self, action, context, timeout="24h"):

        if response == "approved":

# In agent code:
if estimated_cost > 500:

Benefits of Approval Workflows#

  • Safety net against errors or adversarial inputs
  • Audit trail of decisions
  • Gradual trust building (start cautious, relax over time)
  • Compliance (required for many regulated industries)

6. Monitoring and Observability Are Day-One Requirements#

What beginners do: Run agents and hope for the best

What power users do: Instrument everything

What to Monitor#

1. Performance Metrics

  • Token usage per session
  • Response latency
  • API error rates
  • Tool execution success rates

2. Behavioral Metrics

  • Actions taken by type (reads vs writes vs execs)
  • Approval request frequency
  • Error patterns
  • Context window utilization

3. Business Metrics

  • Tasks completed successfully
  • Customer satisfaction (if applicable)
  • Time/cost saved
  • ROI tracking

7. Cost Optimization Matters More Than You Think#

What beginners think: “API costs are negligible”

What power users discover: Token costs add up fast

Our Cost Journey#

Month 1: $45 (playing around, low usage)
Month 2: $230 (going live with customers)
Month 3: $890 (unoptimized, context window bloat)
Month 4: $180 (after optimization)

What we changed:

  • Session scoping (fresh context, fewer wasted tokens)
  • Structured outputs (less verbose responses)
  • Caching frequent queries (vector DB for common questions)
  • Cheaper models for simple tasks (GPT-3.5 for FAQs, Claude for complex reasoning)
  • Prompt compression (remove unnecessary verbosity)

Cost Optimization Tactics#

1. Model Selection by Task

  • Simple tasks: Use cheaper/smaller models
  • Complex reasoning: Use premium models
  • Known answers: Use caching/lookup (zero cost)

2. Context Window Management

  • Don’t include entire conversation history every time
  • Summarize old context
  • Use retrieval instead of stuffing prompts

3. Batch Processing

  • Group similar requests
  • Amortize setup costs across multiple queries

4. Smart Caching

  • Cache LLM responses for common queries
  • TTL-based invalidation
  • Saves 30-40% on our system

Cost Monitoring Dashboard#

We track cost per:

  • Agent
  • Session type
  • Tool invocation
  • Customer interaction

This lets us identify and fix expensive patterns quickly.

Bonus: The Meta-Lesson#

Beginners expect it to “just work” out of the box. Power users understand they’re building a custom system:

  • You design your workflows

This is a feature, not a bug. The flexibility is what makes OpenClaw powerful.
