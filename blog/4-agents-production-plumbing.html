Why We Built This

In 2024, I hit a wall. My plumbing business was growing, but I couldn't keep up. I was drowning in:

  • Customer calls and texts at all hours
  • Scheduling conflicts and double-bookings
  • Research questions ("What's the best tankless water heater for a 3-bath house?")
  • Administrative work‚Äîinvoices, follow-ups, ordering parts

The obvious answer: hire more people. But that's expensive, hard to manage, and I'd still be the bottleneck for decisions.

Then OpenClaw hit my Twitter feed. An AI assistant that could do things‚Äînot just answer questions. It could send messages, search the web, run scripts, integrate with tools. I thought: What if I could automate the coordination layer of my business?

So I built it. And it worked. Too well, actually‚Äîwhich is when I realized how insecure it was.

The Architecture: How It All Fits Together

Our system runs on a $40/month DigitalOcean droplet (4GB RAM, 2 vCPUs). Here's the stack:

  • OpenClaw Gateway running in Docker with network isolation
  • PostgreSQL database for scheduling, customer data, and memory
  • Redis for session management and queues
  • Nginx reverse proxy with rate limiting and origin validation
  • HashiCorp Vault for secrets management (API keys, database credentials)

Each agent runs in its own container with:

  • Read-only filesystem mounts (except for designated workspace directories)
  • Network policies restricting which services they can reach
  • Resource limits (CPU, memory) to prevent runaway processes
  • Separate logging for audit trails

üí° Why This Matters

If one agent is compromised, it can't touch the others. If Huginn's browser tool is exploited, the attacker can't access Muninn's customer database or ZEKE's messaging capabilities.

Security Hardening: What We Did

1. Tool Permission Scoping

Default OpenClaw gives agents near-unlimited tool access. We built a custom permission layer:

  • ZEKE: Can send messages, read/write calendar, query customer database. Cannot exec, cannot write files outside workspace.
  • Huginn: Can browse whitelisted domains, search web, write research files. Cannot exec, cannot access database directly.
  • Muninn: Can read/write memory database, organize files. No network tools, no exec.
  • Odin: Can spawn subagents, run analysis, write plans. High-risk actions require approval.

2. Network Segmentation

We use Docker networks to isolate components:

  • Public network: Only Nginx reverse proxy
  • App network: OpenClaw gateway, Redis
  • Data network: PostgreSQL, Vault

Agents can't talk directly to the database or Vault‚Äîthey go through the gateway, which enforces permissions.

3. Secret Rotation

Every 90 days, we rotate:

  • Anthropic API keys
  • Database passwords
  • Telegram bot tokens
  • SMS provider credentials

We have a script that updates Vault, restarts services, and verifies everything still works. It's tedious, but it's necessary.

4. Monitoring and Alerts

We monitor for:

  • Unusual API usage‚Äîspikes in token consumption
  • Failed auth attempts‚Äîsomeone trying to brute-force the gateway
  • Unexpected tool calls‚Äîe.g., if Muninn suddenly tries to exec a command
  • Database access patterns‚Äîlarge exports or deletions

Alerts go to my phone. If something's wrong, I know within minutes.

5. Staged Updates

OpenClaw updates frequently and often breaks things. We never update production directly. Instead:

  1. Pull the latest version to a dev environment
  2. Run our test suite (scheduled messages, customer interactions, research tasks)
  3. If it passes, deploy to production during low-traffic hours
  4. Monitor for 24 hours before considering it stable

‚ö†Ô∏è Lesson Learned the Hard Way

We once updated OpenClaw during business hours. A breaking change killed ZEKE's message handler. We missed 6 customer inquiries before I caught it. Now we only update on Sundays at 6 AM.

What We've Learned

1. AI Agents Are Production-Ready (With Proper Security)

The narrative that "AI agents are too risky for production" is wrong. The truth is: insecure AI agents are too risky for production. Secure them properly, and they're transformative.

2. You Don't Need to Be a Tech Company

I'm a plumber. I taught myself Docker, learned enough Linux to be dangerous, and figured out OpenClaw by reading docs and asking ChatGPT. If I can do it, you can too. (Or you can hire someone. That's also valid.)

3. Start Small, Then Scale

We started with just ZEKE handling Telegram messages. Once that was stable, we added Huginn for research. Then Muninn for memory. Then Odin for planning. Trying to build all 4 at once would've been overwhelming.

4. Security Can't Be an Afterthought

Our first deployment had no container isolation, plain-text API keys in config files, and ZEKE running with full filesystem access. It worked, but it was a disaster waiting to happen. We got lucky‚Äîwe hardened it before a breach. Don't count on luck.

5. Monitoring Is Worth the Effort

The first time our alert system caught an unusual spike in API usage (turned out to be a bug causing infinite loops), it paid for itself. You need visibility into what your agents are doing.

Would We Do It Again?

Absolutely. This system has transformed how we operate. I spend less time on coordination and more time on actual plumbing. Customer satisfaction is up because response times are faster. And I sleep better knowing that if a customer texts at 2 AM about a burst pipe, ZEKE will triage it and wake me if it's urgent.

But I wouldn't deploy it the way we did initially. If I were starting over, I'd either:

  • Hire someone to set up the hardened architecture from day one, or
  • Use a security-focused fork like NanoClaw to avoid the hardening work

The 3 months we spent hardening our OpenClaw setup was time I could've spent running my business. But the result‚Äîa secure, reliable, production AI system‚Äîwas worth it.
